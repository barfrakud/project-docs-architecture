# Private to Public Promotion Guide

## Input
Private material may include:
- plans,
- prompts,
- implementation reports,
- manuals,
- decisions,
- concept notes.

## Transform rules
When converting private documentation into public documentation:

### Remove
- raw learning notes,
- uncertainty that does not matter publicly,
- discarded experiments,
- repetitive scratch detail,
- overly detailed internal procedure steps,
- secrets or environment-specific sensitive details.

### Keep
- project goal,
- architecture,
- setup instructions,
- CI/CD explanation,
- deployment model,
- stable operational guidance,
- meaningful decisions,
- concise results.

## Tone
Public documentation should be:
- concise,
- clean,
- confident,
- suitable for GitHub readers and recruiters.
