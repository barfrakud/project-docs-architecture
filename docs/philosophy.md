# Documentation Philosophy

## Goal

Documentation in this model serves several roles at once:

1. it documents the work,
2. it supports learning,
3. it improves AI-assisted execution,
4. it becomes the source for later public documentation.

## Main principle

> First create private documentation. Later derive public documentation from it.

This avoids two common failures:
- public documentation that is too raw and chaotic,
- private working knowledge that is lost after implementation.

## What stays private

Private materials may include:
- learning notes,
- rough implementation reasoning,
- prompts,
- exploratory plans,
- rejected ideas,
- concept explanations written for the author's own learning.

## What becomes public

Public materials should include:
- overview,
- architecture,
- setup,
- CI/CD,
- deployment,
- usage,
- selected decisions,
- stable conclusions.

## Benefits

- better recall of what was done,
- better reuse of prompts,
- better onboarding for future-you,
- easier GitHub publication,
- cleaner portfolio presentation.
